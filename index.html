1. COMPOSITION

Composition is a design technique where one class contains a reference to another class as a field. This represents a has-a relationship.

For example, a Car has an Engine, so the Car class will have an Engine object inside it.

It’s a strong form of association where the lifecycle of the contained object is dependent on the container object.

Composition is preferred over inheritance in many cases because it provides better flexibility and loose coupling.

2. ASSOCIATION

Association is a general relationship between two classes that are connected logically but have independent lifecycles.

It represents a ‘uses-a’ or ‘has-a’ relationship.

There are four types of association — one-to-one, one-to-many, many-to-one, and many-to-many.

For example, a Teacher and a Student are associated with each other — a Teacher can teach many students and vice versa.

Association is the umbrella concept under which aggregation and composition fall.

  • 🔹 One-to-One: One person has one passport
  • 🔸 One-to-Many: One teacher teaches many students
  • 🔸 Many-to-One: Many students have one teacher
  • 🔹 Many-to-Many: Many teachers teach many students

3. ENCAPSULATION

Encapsulation is the process of hiding the internal state of an object and restricting access to it by using getter and setter methods.It allows us to protect the data and expose only what’s necessary. —- We achieve encapsulation by declaring class variables as private and providing public methods to access or update them. — This improves security, maintainability, and flexibility of code.

4. ABSTRACTION

Abstraction in Java is the process of hiding the implementation details and exposing only the necessary functionalities to the user. — It helps in reducing complexity and increasing code reusability. — We achieve abstraction using abstract classes and interfaces.

For example, in my project, I used interfaces to define service contracts between layers — so that the implementation could vary without affecting consumers.

17. LinkedHashSet

A LinkedHashSet is a Set that stores unique elements and also remembers the order in which elements were added.

  • Inheritance → LinkedHashSet extends HashSet and internally uses a HashMap + doubly linked list.
  • Uniqueness → Because of Hashing, duplicates are not allowed.
  • Ordering → Maintains insertion order with the help of a linked list that connects entries. Order preserved, no duplicate Alice
  • Complexity → Operations like add(), remove(), contains() are usually O(1) (because of hashing).
  • Use Case → When we need a Set that prevents duplicates but also keeps the insertion order.
  • Thread safety → It is not thread safe. And to make thread safe we need to do Collections.synchronizedSet(new LinkedHashSet<>())

Default Capacity

Default capacity: 16

Load factor: 0.75

Default threshold: 12 (16 × 0.75)

  • Thread safety → It is not thread safe. And to make thread safe we need to do Collections.synchronizedSortedSet(new TreeSet<>())

20. TreeMap

A TreeMap in Java is a map that stores key–value pairs in a sorted order of keys (ascending by default).

  • Inheritance → Implements NavigableMap, backed by a Red-Black Tree.
  • Ordering → Keys are sorted by:
    • Natural ordering (if keys implement Comparable), OR

  • Uniqueness → Keys must be unique (like any Map).

21. ArrayList

An ArrayList in Java is a resizable array that can grow or shrink dynamically. It maintains insertion order and allows duplicate elements.

  • Package → java.util
  • Internally → Uses a dynamic array.
  • Default Capacity → In Java 8+, starts with 0, becomes 10 when first element is added.
  • Growth Mechanism → When full, new capacity = oldCapacity × 1.5.
  • Ordering → Maintains insertion order, supports random access (get(index) is O(1)).
  • Nulls → Multiple nulls allowed.
  • Performance:
    • get(index) → O(1)
    • add() → Amortized O(1)
    • remove() → O(n) (because shifting needed)
  • Use Case → Best when you need fast random access and frequent read operations, but not ideal for lots of insertions/removals in the middle.

Difference with LinkedList (Interview Point)

In Java 8 and later, an ArrayList starts with an initial capacity of 0, but when the first element is added, it becomes 10. Whenever it gets full, it increases its capacity by 1.5 times the old capacity.

25. Hashtable

A Hashtable in Java is a legacy class (before Java Collections Framework) that stores key-value pairs in a hash table data structure. It does not allow null keys or null values and is synchronized (thread-safe).

  • Package → java.util
  • Internally → Uses an array of buckets where each bucket stores a linked list of key-value pairs (similar to HashMap but older).
  • Default Capacity → 11 (initial), grows by 2*oldCapacity + 1 when rehashed.
  • Load Factor → 0.75 (default).
  • Performance:
    • put() → Average O(1), Worst O(n).
    • get() → Average O(1), Worst O(n).
  • Nulls → Not allowed (neither key nor value).
  • Thread Safety → Methods are synchronized → multiple threads can safely access, but slower than HashMap.
  • Use Case → When you specifically need a thread-safe map in legacy code. For modern apps, prefer ConcurrentHashMap.

HashMap vs Hashtable

HashMap is modern and faster, but it is not thread-safe. Hashtable is old, synchronized (thread-safe), but slower. Hashtable also does not allow null keys or null values. In today’s applications, if we need thread safety, we use ConcurrentHashMap instead.

32. OutOfMemoryError

An OutOfMemoryError in Java happens when the JVM cannot allocate more memory for objects because the heap, metaspace, or other memory area is full and garbage collection cannot free enough space.

Common causes:

  • Loading massive datasets (DB records, files, API responses) into memory at once.
  • Infinite object creation without release.
  • Memory leaks (objects still referenced but not needed).
  • Large caches that keep growing.
  • More concurrent requests → more objects created in memory at the same time.
  • If requests involve heavy processing or big data loads, memory fills up faster.
  • If code holds onto objects too long (e.g., session data, caches), the growth is multiplied.
  • Garbage Collector may not be able to keep up → heap fills → OOM.

Types (most common):

  • java.lang.OutOfMemoryError: Java heap space → Heap is full.
  • java.lang.OutOfMemoryError: Metaspace → Class metadata space is full.
  • java.lang.OutOfMemoryError: GC overhead limit exceeded → GC runs too much but still can’t free memory.

Example Scenario:

  • Service reads a 1GB file into a single String.
  • Or a loop keeps adding objects into a List without clearing.
  • Or loading millions of records from DB into memory at once.
  • You have a REST API that fetches orders from DB in one go for each user.
  • With 100 users, memory usage is fine.
  • Suddenly, traffic spikes to 5,000 users at the same time.
  • Now thousands of List<Order> objects are in heap at once → memory gets full → OOM: Java heap space.

Fix / Prevention

  1. Pagination / Streaming – Load data in batches, not all at once.
  2. Rate Limiting / Backpressure – Limit concurrent heavy requests.
  3. Horizontal Scaling – Multiple service instances with load balancer.
  4. Fetch Only Required Fields – Use DTO, avoid full entity load.
  5. Async Processing – Use Kafka/RabbitMQ for heavy processing.
  6. Close Resources – Avoid memory leaks.
  7. Heap Size Tuning – Increase with -Xmx after fixing code issues.

34. Memory Leak Management

Memory Leak happens when objects are no longer needed but still referenced in memory, so Garbage Collector cannot remove them. — Over time, memory usage keeps growing until OutOfMemoryError occurs. —

Fix / Prevention

  1. Close Resources Properly
    • Use try-with-resources for DB, file, socket handling.
  2. Remove References When Done
    • Set objects to null if they are no longer needed.
  3. Use WeakReference
    • For caches, so GC can clear unused objects.
  4. Unregister Listeners
    • Remove listeners once they are no longer needed.
  5. ThreadLocal Cleanup
    • Call remove() on ThreadLocal variables after use.
  6. Cache with Eviction
    • Use LRU or TTL to auto-remove old data.

35. Circuit Breaker

Circuit Breaker is a resilience pattern that *stops calling a failing remote service* so your system doesn’t get overwhelmed and failures don’t cascade. —- Circuit Breaker is a resilience pattern that prevents cascading failures by stopping calls to a repeatedly failing downstream service. It has three states — closed, open, half‑open — and uses thresholds ( failure rate, minimum calls, sliding window ) to decide when to open. When open it fails fast or uses fallback, then periodically tests the downstream before closing again.

35.1 CIRCUIT BREAKER EXAMPLE :

Problem (no circuit breaker):

Service A → calls B → B calls C. If C goes down or becomes very slow:

  • B’s threads/connection pool wait for C (timeouts), threads get stuck.
  • B cannot serve other requests → B becomes slow/unavailable.
  • A sees timeouts/errors from B → A fails too → cascading failure across system.

With Circuit Breaker:

  • B detects failures to C, opens its CB for calls to C.
  • Instead of blocking threads waiting for C, B immediately returns fallback (or error) — threads freed quickly.
  • B stays healthy and can still serve other functionality. The cascade stops.

36. Kafka

Kafka is a distributed Event streaming platform I’ve used to connect and decouple microservices.It helps handle traffic spikes smoothly and process data in real time. —- I usually create topics and partition them so multiple consumers can read in parallel.In a consumer group, each partition is handled by only one consumer for efficiency.I track consumer lag to ensure messages are being processed on time. — By default, Kafka gives at-least-once delivery — I’ve used idempotent or transactional producers when exactly-once was needed.

36.1 What is Event Streaming?

Event streaming means continuously capturing, storing, and processing streams of events (small pieces of data) in real time as they happen. — These "events" can be anything that happens in a system — like a new order placed, a payment completed, a sensor reading, or a user clicking a button. — Event streaming means data flows continuously like a stream instead of being stored and processed in batches.

What it means

A Distributed Event Streaming Platform is a system designed to:

  1. Capture events (data changes, actions, messages) in real-time.
  2. Store them durably so they can be read later.
  3. Process and deliver them to other systems or services.

Distributed means the platform is spread across multiple servers/nodes, not running on a single machine.

This gives:

  • Scalability → handle more events by adding more nodes.
  • Fault tolerance → if one node fails, others take over.
  • High availability → no downtime for producers and consumers.

36.2 Why Kafka? (Value)

  • Decouple services (loose coupling).
  • Handle spikes with buffering; smooth traffic.
  • Scalable via partitions; durable via replication.
  • Replayable: reprocess by seeking older offsets.
  • Low latency streaming + exactly-once processing (with the right config).

Kafka Real-time example

Imagine an e-commerce site:

  1. User places an order → event is sent to Kafka’s "Orders" topic.
  2. This event is stored across multiple partitions/nodes.
  3. Inventory Service consumes the event → reduces stock.
  4. Payment Service consumes the event → charges the customer.
  5. Notification Service consumes the event → sends confirmation email.

All these services work independently and in parallel because Kafka is distributed and can stream events to all of them simultaneously.

37. Saga Pattern – Backend World

A Saga Pattern is a design pattern used in microservices to manage distributed transactions without using a global transaction lock. — Instead of one big transaction, it breaks it into multiple local transactions across services, each with a compensating action in case of failure.

Hinglish:

Suppose tumhare paas ek e-commerce system hai, aur ek order place karte time tumhe multiple services call karni hain:

  1. Payment Service – Payment deduct karo
  2. Inventory Service – Stock reduce karo
  3. Shipping Service – Shipment create karo

Agar step 2 fail ho jaye, tumhe step 1 ka reverse karna hoga (refund payment). Ye sequence + rollback ka plan hi Saga Pattern hai.

Flow Types in Backend:

  • Choreography – Services communicate via events (no central controller).
  • Orchestration – A central "orchestrator" service decides the flow.

37.2 Saga Pattern – Frontend (React + Redux-Saga)

In our React frontend, Saga Pattern ensures that when a user performs an action involving multiple backend services, each service completes its part independently, and if any step fails, compensating actions roll back the previous steps to keep data consistent.

In React, Redux-Saga is a middleware that handles side effects (async tasks like API calls, websocket communication, delays, retries) in a predictable way. It’s called "Saga" because it also organizes async operations in steps, and can handle rollback/retry logic just like backend sagas.

Why Use Saga (Frontend + Backend)

English:

  • Keeps async flow predictable.
  • Allows retries, delays, rollbacks.
  • Decouples business logic from UI.
  • Easier to test complex async workflows.

Real-Time Full Example (Combined)

Scenario:

You are building an Order Checkout in React that talks to microservices.

Steps:

  1. Frontend calls Payment API (Payment Service).
  2. If payment succeeds → call Inventory API (Inventory Service).
  3. If inventory is available → call Shipping API (Shipping Service).
  4. If any step fails → call compensation APIs (Refund Payment, Restock Inventory).

Distributed Transaction?

A distributed transaction is when a single business process spans multiple services or databases and we need all-or-nothing behavior. Traditionally, we used XA/2PC, where a coordinator ensures commit or rollback across participants, but it’s slow and doesn’t fit microservices well. In modern systems, we use the Saga pattern, where each service does a local transaction and publishes events. If a later step fails, we trigger compensating actions to undo previous steps. For example, in a travel booking app, if the car rental fails after booking flight and hotel, we send events to cancel both, keeping the system consistent.

Approaches to Handle Distributed Transactions

A. XA / Two-Phase Commit (2PC)

  • What: Traditional approach for distributed transactions in relational databases.
  • How:
    1. Prepare phase → Coordinator asks all participants if they can commit.
    2. Commit phase → If all say "yes", coordinator sends commit; if any says "no", sends rollback.
  • Pros: Strong consistency.
  • Cons: Slow, blocking, not very cloud/microservices-friendly.

Example: Booking a flight + hotel in one step, both in different databases. If hotel booking fails, flight booking rolls back.

B. Saga Pattern

  • What: A sequence of local transactions with compensating transactions for rollback.
  • How:
    • Choreography: Services communicate via events.
    • Orchestration: A central orchestrator tells each service what to do.
  • Pros: Works well in microservices, scalable.
  • Cons: Eventual consistency, more complex to implement.
